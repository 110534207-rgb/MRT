# 系統分析

## 1. 使用者需求

本節根據需求規格文件，詳細描述系統需要滿足的功能性與非功能性需求。

### 1.1 功能性需求

| 角色 | 功能 | 描述 |
| :--- | :--- | :--- |
| **訪客** | 瀏覽路線圖 | 使用者可以查看各捷運線路的互動式地圖，地圖上會標示所有站點。 |
| | 查詢景點 | 使用者點擊任一站點後，系統應顯示該站點附近的推薦景點列表。 |
| **管理員** | 管理員登入 | 使用者可透過點擊「管理員登入」按鈕並輸入正確密碼，進入管理模式。 |
| | 新增景點 | 登入後，管理員可以為指定的捷運站點增加新的景點。 |
| | 刪除景點 | 登入後，管理員可以刪除指定捷運站點的現有景點。 |

### 1.2 非功能性需求

| 需求類型 | 描述 |
| :--- | :--- |
| **相容性** | 網站必須能夠在主流的電腦（Chrome, Firefox, Edge）和手機（Safari, Chrome）瀏覽器上正常運作。 |
| **易用性** | 介面應簡潔直觀，訪客能輕鬆上手查詢，管理員也能方便地進行內容管理。 |
| **安全性** | 管理員功能必須透過密碼驗證才能存取，防止未授權的操作。 |
| **部署** | 專案的最終版本檔案需存放於 `110534207` 目錄下。 |

## 2. 資料流程圖 (Data Flow Diagram)

由於本專案為純前端應用，所有資料皆儲存於客戶端（瀏覽器）的 JavaScript 變數中，不涉及後端伺服器或資料庫。

### 2.1 訪客操作流程

1.  **請求頁面**：訪客在瀏覽器中開啟任一捷運線路的 `.html` 檔案。
2.  **載入資料**：
    *   系統（瀏覽器）解析 HTML 並執行內嵌的 JavaScript。
    *   JavaScript 從 `getDefaultStations` 函式中讀取預先定義好的車站與景點資料陣列。
    *   系統根據資料動態產生捷運路線圖與站點。
3.  **查詢景點**：
    *   訪客點擊地圖上的任一站點。
    *   系統觸發點擊事件，從 JavaScript 陣列中篩選出對應站點的景點資料。
    *   系統將景點資料呈現在畫面上的一個彈出視窗或指定區域中。

### 2.2 管理員操作流程

1.  **進入管理模式**：
    *   管理員點擊「管理員登入」按鈕。
    *   系統彈出一個提示框（`prompt`）要求輸入密碼。
    *   管理員輸入密碼 `123`。
    *   系統驗證密碼，成功後進入管理模式，並在介面上顯示「新增」與「刪除」按鈕。
2.  **新增景點**：
    *   管理員點擊特定站點旁的「新增景點」按鈕。
    *   系統彈出提示框，要求輸入要新增的景點名稱。
    *   管理員輸入名稱並確認。
    *   系統透過 JavaScript 將新的景點資料加入對應車站的景點陣列中。
    *   系統即時更新畫面，顯示包含新景點的列表。
3.  **刪除景點**：
    *   管理員點擊某個景點旁的「刪除」按鈕。
    *   系統透過 JavaScript 從對應車站的景點陣列中移除該筆資料。
    *   系統即時更新畫面，移除被刪除的景點。

*註：所有資料的變更（新增/刪除）僅存在於當次頁面瀏覽的生命週期中，重新整理頁面後將會還原成 `getDefaultStations` 函式中的初始狀態。*
